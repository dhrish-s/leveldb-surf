=== STANDARD BENCHMARKS ===

  Benchmark      Bloom (µs/op)   SuRF (µs/op)   Change       Winner
  -----------    -------------   ------------   ----------   ------
  readrandom     2.847           8.977          +215.2%      Bloom
  readseq        0.212           0.447          +110.8%      Bloom
  seekrandom     8.891           6.634          -25.4%       SuRF
  fillrandom     2.638           4.078          +54.6%       Bloom

  Notes:
  - readrandom: point lookups. Bloom's simple bit-check is much cheaper
    than SuRF's trie deserialization. Expected result.
  - readseq: sequential scan. SuRF filter is larger and costlier to load.
    Expected result.
  - seekrandom: random seeks. SuRF is 25.4% FASTER — SuRF's trie structure
    aids seek navigation even without explicit lo/hi bounds.
  - fillrandom: writes. SuRF trie construction is more expensive than
    Bloom bit-array hashing. Expected result.


=== VARIABLE MISS RATE — RANGE SCANS (range_width=10) ===

  Benchmark       Bloom (µs/op)   SuRF (µs/op)   Change      Winner    Keys Scanned
  ------------    -------------   ------------   ---------   ------    ------------
  surfscan100     1.298           1.374          +5.9%       Bloom     0
  surfscan75      3.922           5.005          +27.6%      Bloom     1,736,663
  surfscan50      5.213           4.760          -8.7%       SuRF      3,478,645
  surfscan25      7.214           5.817          -19.4%      SuRF      5,215,152
  surfscan0       7.503           6.934          -7.6%       SuRF      6,952,830

  Key findings:
  1. SuRF advantage emerges at 50% miss rate and grows through 25% miss.
  2. SuRF is fastest relative to Bloom at 25% miss rate (-19.4%).
  3. At 100% miss, both filters are fast (~1.3 µs) because no data blocks
     are read regardless — the difference is within noise.
  4. At 0% miss (all hits), SuRF is still 7.6% faster, indicating the
     SuRF trie aids range iteration even when ranges contain keys.

  Why SuRF wins on hit-heavy workloads:
  When queries land inside the key range, LevelDB must scan data blocks.
  SuRF's compressed trie provides structural information about key
  distribution that helps the iterator navigate more efficiently,
  whereas Bloom provides no structural information at all — it only
  answers yes/no for exact keys.


=== WIDE RANGE SCAN (range_width=100, 100% miss) ===

  --------------   -------------   ------------   --------   ------
  surfscan_wide    1.361           1.411          +3.7%      Bloom

  Note: At 100% miss rate with wide ranges, both filters produce nearly
  identical results (~1.4 µs) because no data blocks are read. The
  range width does not affect performance when all ranges are empty.


=== WRITE PERFORMANCE ===

  Benchmark      Bloom (µs/op)   SuRF (µs/op)   Change      Winner
  -----------    -------------   ------------   ----------   ------
  fillrandom     2.638           4.078          +54.6%       Bloom

  Note: SuRF trie construction during compaction is more expensive than
  Bloom's bit-array hashing. This is the expected trade-off — SuRF pays
  a write-time cost to enable range query filtering at read time.


=== TRADE-OFF SUMMARY ===

  SuRF wins:
  - Range scans with mixed hit/miss workloads (surfscan50: -8.7%, surfscan25: -19.4%)
  - Range scans with all hits (surfscan0: -7.6%)
  - Random seeks (seekrandom: -25.4%)

  Bloom wins:
  - Point lookups (readrandom: +215.2%)
  - Sequential reads (readseq: +110.8%)
  - Write throughput (fillrandom: +54.6%)
  - Pure miss range scans (surfscan100: +5.9% — but both are fast)

  Conclusion:
  The right filter depends on workload. Applications dominated by range
  scans and seeks (time-series databases, analytics, blockchain range
  queries) benefit from SuRF. Applications dominated by point lookups
  and writes (caching, exact-key retrieval) are better served by Bloom.
